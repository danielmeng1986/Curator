# Direct Album Import

## Goal

`/import/albums` imports selected folders directly into the permanent `album` table; it does not create `workspace_album` rows. The flow may reuse the local folder naming convention (for example, `{model_name} in {album_name}`), configured source root, archive destination, and copy/move behavior. The destination is explicitly labeled **Import to permanent albums**.

## Staged Workflow

1. **Choose source** — select one album folder or a directory of album folders. Show configured source root and move/copy choice; moving is the default, and **Keep source files** changes it to copy.
2. **Discover and parse** — build one draft per folder. Parse candidate primary Model and title while retaining the source folder name.
3. **Review metadata** — spreadsheet-like selected drafts. Set Studio, Status, title, Model(s), canonical `path`, optional dates/rating, and optional **Belongs to** logical Album. Model/Studio controls support inline creation.
4. **Validate and preview** — show final album values, entity creations, `album_model`/`album_relation` rows, computed paths, duplicate checks, and filesystem conflicts. Invalid rows cannot be selected.
5. **Confirm and execute** — confirm counts of Albums, Studios, Models, and relationships created/reused plus the filesystem action. Run as a recorded, recoverable operation.
6. **Results** — report success/failure per row, link created Albums, and offer a copyable/downloadable operation summary. Remove successful rows from the draft; retain failures for correction and retry.

## Mapping

| Import draft value | Permanent write |
| --- | --- |
| Selected/created Studio | `album.studio_id` |
| Selected Status | `album.status_id` |
| Album title | `album.title` |
| Source/archive path | `album.path` |
| Primary/additional Models | One `album_model` row per unique selected Model; no Model field is written to `album` |
| Per-model age, role, remarks | Matching `album_model` fields |
| Selected logical/canonical Album | One `album_relation` row with `relation_type = 'BELONGS_TO'` |
| Identity/audit values | Generated by the application |

## Import Rules

- Studios and Models are selected by name or created inline; parsed Model names are suggestions, never automatic final matches.
- Multiple source rows resolving to the same new Studio or Model create it once.
- Primary and additional Models use the same relationship semantics as the Album form: each unique selection creates or reuses one `album_model` row. No Model value is stored on `album` itself.
- Duplicate Model selections inside one draft Album are merged. Each album/model pair is unique.
- An optional logical/canonical Album creates a `BELONGS_TO` relation after both Albums exist. The imported Album cannot be related to itself.
- The initial duplicate rule is same Studio + title, with `album.path` checked independently for collisions. Conflicts block import until an explicit merge workflow exists.
- On filesystem/database failure, the operation must report the affected row precisely and use transaction/compensation behavior so a successful database record is not claimed for an unsuccessful filesystem import.
